Cheat Sheet — Planted Bugs on demo/bugs branch

Three bugs, each a different class. You know them; the audience doesn't.

Bug 1 — Off-by-one in rating validation

routes/reviews.js:15 — rating >= 5 should be rating > 5

Effect: Users can't submit 5-star reviews. The "Excellent" option is silently rejected. Subtle — the form looks like it works for 1-4 stars.

Bug 2 — Swapped parameters in UPDATE query

routes/books.js:85 — description and isbn arguments are swapped in .run()

Effect: Editing a book silently puts the description in the ISBN field and vice versa. Data corruption that looks fine at first glance.

Bug 3 — Wrong sort order on book list

routes/books.js:6 — ORDER BY title DESC instead of ORDER BY created_at DESC

Effect: Book list sorts by title instead of newest-first. Not a crash, but a behavior regression — the commit message says "add sorting" which masks it.

Demo Act 1 — Debugging 15 min

Bug 1: The 5-star problem (5 min)

Show the bug live first — go to a book, try to submit a review with rating 5. It fails with "Rating must be between 1 and 5." Rating 4 works fine.

"Let's ask Claude to figure this out."

Promptusers are reporting they can't submit 5-star reviews. when they select "5 - Excellent" and submit, they get "Rating must be between 1 and 5". ratings 1-4 work fine. find the bug.

What Claude will do: Read the review route, spot rating >= 5, explain the off-by-one, and fix it to rating > 5.

Point out: "I gave Claude the symptom, not the file. I said 'users can't submit 5-star reviews' — not 'check line 15 of reviews.js.' Describe the problem, let Claude find the code."

Teaching moment
"Notice it read the route file, found the validation line, and explained WHY it's wrong — not just WHAT to change. That explanation is the value. A find-and-replace could change the character, but Claude tells you the logic error."

Bug 2: The data corruption (5 min)

"Let's try something harder. I edited a book earlier and something weird happened." Go to a book, click Edit, change the description, save. Go back — the ISBN and description are swapped.

"Hmm. Let me show Claude what I'm seeing."

Promptsomething weird is happening with book editing. I edited the description of a book and saved it. now the ISBN field shows what used to be the description, and the description shows the ISBN. the data seems to be getting swapped somehow. look at the update route.

What Claude will do: Read the update route, compare the SQL column order to the .run() parameter order, and spot the swap.

Point out: "This is a class of bug that's really hard to spot in a code review — the line looks correct at a glance. The column names and the variable names don't match up in the argument list. Claude caught it because it actually traced the data flow."

Bug 3: The behavioral regression (5 min)

"One more. A teammate says the book list used to show newest books first, now it doesn't. Let's see what changed."

Promptthe book list page is supposed to show the most recently added books first, but it's not doing that anymore. something changed. can you check the git history for the book list route and figure out what happened?

What Claude will do: Run git log / git diff on the route file, find the ORDER BY change, and explain the regression.

Point out: "I told Claude to check the git history. That's a key move — when something 'used to work,' the answer is usually in a recent commit. Claude can read diffs and correlate them with behavior changes."

Key takeaway
Three different debugging strategies: (1) give the symptom, let Claude find the cause; (2) describe the unexpected behavior, point at the area; (3) ask Claude to investigate the git history. All three start with WHAT'S WRONG, not WHERE TO LOOK.
Demo Act 2 — Testing 13 min

Say: "We found and fixed three bugs. None of them had tests. Let's fix that — and let's have Claude write the tests."

Step 1 — Write tests for the bug we just fixed (5 min)

Promptwrite tests for the review submission route. cover: valid review (rating 1-5), rejected if rating is out of range, rejected if reviewer name is empty. use the built-in node test runner, no external test frameworks.

What to point out as Claude works:

  • "I said 'no external test frameworks' — Node has a built-in test runner since v18. Claude respects the constraint."
  • "It's setting up the database, seeding test data, and tearing it down. It wrote the scaffolding we didn't want to write."
  • "Look at the test for rating 5 — that's a regression test. If someone reintroduces the off-by-one, this catches it."

Run the tests live:

Terminalnode --test

Show them passing. "We went from zero tests to a safety net in under two minutes."

Step 2 — Ask for broader coverage (5 min)

Promptnow write tests for the book routes — cover creating a book, creating with missing title (should fail), the book detail page returning reviews, and deleting a book also deleting its reviews.

Point out: "I described the test SCENARIOS, not the implementation. 'Creating with missing title should fail' — Claude figures out the assertion."

Run the tests again. Show the full suite passing.

Step 3 — Show the "write a test that fails" trick (3 min)

Promptwrite a test that verifies the book update route puts the ISBN in the isbn column and the description in the description column (not swapped). run it against the current code.

"If we hadn't fixed Bug 2 yet, this test would fail. That's how you use tests to PROVE a bug exists before fixing it. Write the test, see it fail, fix the code, see it pass."

Teaching moment
"The hardest part of writing tests is knowing WHAT to test. You already know that — you just described the behavior. Let Claude handle the HOW."
Demo Act 3 — Code Review 12 min

Say: "Last act. A teammate submitted a PR to add sorting to the book list. Let's review it with Claude."

Terminalgit diff main..demo/pr-review

Show the diff on screen briefly. "Looks reasonable — sort controls, a few query params. Let's see what Claude thinks."

Promptreview the diff between main and demo/pr-review. look for bugs, security issues, and anything you'd flag in a code review.

What Claude should find:

  • SQL injection — ${sort} and ${order} are interpolated directly from req.query into the SQL string. An attacker can pass arbitrary SQL via the URL. This is the big one.
  • No input validation — sort and order accept any string. Should be allowlisted to valid column names and ASC/DESC.
  • No tests — new feature with no test coverage.
The SQL injection moment
Pause on this one. "This is a real security vulnerability. The sort parameter goes straight into the query. Someone could visit /books?sort=title;DROP TABLE books--. In a real app, this is a CVE." Show the audience the vulnerable line. Let it sink in.

Ask Claude to fix it:

Promptfix the SQL injection in the sort feature. allowlist the valid sort columns and only accept ASC or DESC for the order.

What Claude will do: Add a whitelist of valid columns, validate the order param, and use parameterized queries or string matching instead of interpolation.

Point out:

  • "Claude didn't just find the bug — it explained the attack vector. That context is what makes this more useful than a linter."
  • "The fix is an allowlist, not a blocklist. Claude chose the right security pattern."
  • "In a real workflow, you'd run /review on a PR branch and get this kind of feedback before merge."
Key takeaway
Claude Code review catches things at different altitudes: logic errors (the off-by-one), security vulnerabilities (SQL injection), missing practices (no tests). It's not replacing human review — it's a first pass that catches the things humans skim past.
Talk Wrap-up 3 min

Debugging:

  • Describe the symptom, not the location
  • When something "used to work" — ask Claude to check git history
  • Claude traces data flow, which catches bugs that look correct at a glance

Testing:

  • Describe the scenarios, let Claude write the implementation
  • "Write a test that fails" is a powerful pattern for proving bugs
  • Going from zero to a test suite takes minutes, not hours

Code review:

  • Claude catches security issues, logic errors, and missing practices in one pass
  • It explains the WHY — not just "this line is wrong" but "here's the attack vector"
  • Use it as a first pass before human review, not a replacement for it
