The Demo Repo: BookBuddy

A minimal Express.js book review app. Simple enough to understand in 2 minutes, complex enough for multi-file demos. Pre-planted with refactoring opportunities.

bookbuddy/ ├── package.json ├── server.js # Express app entry ├── models/ │ ├── book.js # Book schema (title, author, isbn) │ └── review.js # Review schema (text, rating) ├── routes/ │ ├── books.js # CRUD for books │ └── reviews.js # CRUD for reviews ├── views/ │ ├── layout.ejs # Base template │ ├── book-list.ejs # List page │ └── book-detail.ejs # Detail + reviews ├── public/ │ └── style.css └── seed.js # Populate sample data

Pre-planted issues for demos:

  • Database queries inlined in route handlers (extractable)
  • Inconsistent error handling (some try/catch, some not)
  • No search or filtering (feature to add)
  • Duplicated validation logic between routes
  • Snake_case in some places, camelCase in others
Prep note
Use SQLite via better-sqlite3 so there's no DB server to set up. Attendees clone, npm install, npm start, done.

Everyone does:

  • Clone the BookBuddy repo
  • npm install && npm run seed && npm start
  • Open http://localhost:3000 — see the app running
  • Open a second terminal in the repo directory, run claude

Sanity check prompt — everyone types this:

Promptwhat does this app do? look at the code and give me a 2-sentence summary

Point out: Claude read the files, understood the structure, answered in context. That's the baseline — it already knows your code.

Demo Your First Multi-File Edit 12 min

Setup: "Let's add a feature that doesn't exist yet — book categories. A book should have a genre, users should be able to filter by genre. Watch how many files Claude touches."

Promptadd a "genre" field to books. it should be a string, one of: fiction, non-fiction, science, history, biography. add a dropdown filter on the book list page so users can filter by genre. update the seed data to include genres.

What to point out as Claude works:

  • It reads first. Watch it read the model, routes, and views before editing anything. "It's building a mental map."
  • It edits multiple files in sequence. Model → route → view → seed data. "One prompt, four files changed."
  • It maintains consistency. The genre values in the seed data match the dropdown options. The route handler filters correctly.

After it finishes: Refresh the browser. Show the filter working. "That would've been 15 minutes of jumping between files. Claude did it in one shot because it could see the whole picture."

Teaching moment
If Claude asks a clarifying question ("should genre be required?"), let it happen. Tell the audience: "This is good — it's asking instead of guessing. Answer it and move on."

Choose one:

Option A — Searchadd a search bar to the book list page. users should be able to search by title or author. the search should work as a form submission, not ajax.

Demo Refactoring Across a Codebase 10 min

Setup: "Now let's clean up. The route files have raw SQL queries mixed in with request handling. Let's extract that."

Promptthe route handlers in routes/books.js and routes/reviews.js have database queries inlined. extract all database operations into a db/ directory with separate files for book queries and review queries. the routes should import and call those functions instead.

What to point out:

  • It creates new files and updates existing ones. New db/books.js and db/reviews.js, modified route files.
  • It names things sensibly. Functions like getBookById, getAllBooks — matches the pattern.
  • The app still works. Refresh the browser. Same behavior, cleaner code.

Follow-up — show a rename refactor:

Promptthere's inconsistent naming in the codebase — some variables use snake_case and some use camelCase. normalize everything to camelCase.

"This is the kind of tedious refactoring that takes 30 minutes of find-and-replace. Claude does it in seconds and catches the edge cases you'd miss."

Pitfall to demo
If the codebase is small, Claude will do this perfectly. In a large codebase, it might miss some files. That's a real limitation — always verify refactors. Show grep -r "snake_case_pattern" . to check.
Exercise Refactor Challenge 7 min

Say: "Try a refactor. Here are two options — or come up with your own."

Option A — Error handlingthe error handling in the routes is inconsistent. some handlers have try/catch, some don't. add consistent error handling to all route handlers — if a database operation fails, return a 500 with a user-friendly error page.
Option B — Validationthere's duplicated validation logic in the book and review routes. extract a shared validation middleware that both routes can use.

Key observation to share after: "Notice how the prompt didn't say which files to change or how to structure it. You described the outcome, Claude figured out the implementation. That's the sweet spot."

Demo: Breaking up big tasks

"If you ask Claude to 'add a full user authentication system with OAuth, roles, permissions, and an admin panel,' it'll try to do it all at once and probably get confused. Instead:"

Better approachlet's add user auth. start with just the basics: a users table, a registration form, and a login form with session-based auth. no OAuth, no roles yet — just email and password.

"Small, concrete tasks. You can always ask for the next piece after."

Rule of thumb
If your prompt is more than 3-4 sentences, you're probably asking for too much at once. Break it up.
Talk Common Pitfalls 5 min
Pitfall 1 — The essay prompt
"I want you to add a search feature. The search should support full-text search across title and author fields. It should be case-insensitive. The results should be paginated with 10 results per page. The search bar should have autocomplete. Use debouncing on the input..."

Better: "add search to the book list — search by title or author." Start simple. Add requirements after you see the first result.
Pitfall 2 — Not verifying
Claude says it's done. The code looks right. But did you actually run it? Always test. Refresh the browser, click the thing, check the edge cases. Claude is confident even when it's wrong.
Pitfall 3 — The gold-plating spiral
You ask for a simple feature. Claude adds error handling, input validation, loading states, accessibility attributes, and a test suite. Sometimes that's great. Sometimes you just wanted a button. If Claude over-builds, tell it: "simpler — I just need X."
Pitfall 4 — The "just fix it" loop
Something breaks. You say "fix it." Claude changes something. Still broken. "Fix it." Another change. Worse now. Stop and describe the actual problem: "when I click submit, I get a 500 error. The console says 'Cannot read property id of undefined' on line 34 of routes/books.js."

Summary slide:

  • Describe outcomes, not implementations
  • Start small, iterate
  • Always verify in the browser / terminal
  • Use /compact between tasks
  • Put project conventions in CLAUDE.md
  • When stuck, give Claude the error message, not "fix it"
Talk Q&A 3 min

Open floor. Common questions to have answers ready for:

  • "Does it work with [language/framework]?" — Yes, it works with anything. It's especially strong with well-documented languages and frameworks.
  • "Can it access the internet / APIs?" — Not by default. It works with your local files. You can add MCP servers for external integrations.
  • "Is my code sent to Anthropic?" — Yes, the code it reads is sent to the API. Check your company's policy. The API doesn't train on your data.
  • "How is this different from Copilot?" — Copilot autocompletes as you type. Claude Code works at a higher level — you describe what you want, it plans and executes across files.
