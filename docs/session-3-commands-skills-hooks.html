Claude Code Summit

Commands, Skills, and Hooks

Tucker Kidd 15:00 – 16:00 Workshop

The Three Customization Layers

Commands

Reusable prompts you invoke with /name. Like shell aliases for Claude. Live in .claude/commands/.

Skills

Persistent context Claude reads automatically when relevant. Your project's rules and patterns. Live in .claude/skills/.

Hooks

Shell commands that fire on events (file edited, tool called). Automated guardrails. Live in .claude/settings.json.

Talk The Mental Model 5 min

Say: "In the earlier sessions you typed prompts from scratch every time. That works, but you end up repeating yourself — same instructions, same conventions, same checks. This session is about making Claude remember your workflow."

The three layers, one sentence each:

  • Commands — "Do this thing I keep asking for." A saved prompt you trigger with a slash.
  • Skills — "Here are the rules." Context Claude reads automatically so you don't have to repeat your standards.
  • Hooks — "Run this check every time." Shell commands that fire on events, no human in the loop.

Where they live:

.claude/ ├── commands/ # markdown files → /command-name │ ├── add-route.md │ └── preflight.md ├── skills/ # markdown files → auto-loaded context │ ├── coding-standards.md │ └── security-checklist.md └── settings.json # hooks config

"All three are just files in your repo. They get committed, shared with your team, and versioned like any other code. That's the key insight — your Claude workflow is codified, not tribal knowledge."

Framing
"Think of it like this: CLAUDE.md is your project README for Claude. Commands are your shell aliases. Skills are your team's style guide. Hooks are your CI checks, but local."
Demo Build a Slash Command 10 min

Say: "Let's build a command from scratch. I find myself constantly asking Claude to explain how a request flows through the app. Let's turn that into a one-word command."

Step 1 — Create the file (do this live in the terminal, not in Claude):

Terminalmkdir -p .claude/commands

Now create the command file. Open it in an editor or use Claude:

Promptcreate a file at .claude/commands/explain.md with this content: Explain how the request lifecycle works for $ARGUMENTS. Trace the full path from the HTTP request to the response: 1. Which route file handles it? 2. What middleware runs? 3. What database queries execute? 4. Which view template renders? 5. What data gets passed to the template? Keep it concise — one paragraph per step. Use actual file paths and line numbers.

Point out two things:

  • "The file name becomes the command name. explain.md → /explain."
  • "$ARGUMENTS is a placeholder — whatever you type after the command gets inserted there."

Step 2 — Use it:

Type in Claude Code/explain GET /books/3

Claude traces the full request path — route file, query, view template, data passed. "One slash command instead of typing that whole instruction every time."

Try another/explain POST /reviews

"Same command, different input. It adapts."

Step 3 — Build a more practical command: /preflight

File: .claude/commands/preflight.mdRun the pre-commit checklist before I commit: 1. Run the test suite with `node --test` and report any failures 2. Check for console.log statements that look like debugging leftovers (not the server startup log) 3. Grep for TODO or FIXME comments and list them 4. Run `git diff --staged` and summarize what's about to be committed 5. If everything passes, say "Ready to commit." If not, list what needs fixing.

Create it live, then run it:

Type in Claude Code/preflight

"This is a workflow in a file. Your team can share it. New developers get the same pre-commit checks on day one."

Key idea
Commands are prompts you've promoted from one-off to reusable. If you've typed the same kind of instruction three times, it should be a command.
Exercise Write Your Own Command 8 min

Say: "Your turn. Create a command and use it. Here are some ideas, or make up your own."

Option A — /add-routeCreate .claude/commands/add-route.md Add a new Express route to BookBuddy. Create a route file at routes/$ARGUMENTS.js with CRUD handlers (GET list, GET detail, POST create, POST delete). Register it in server.js. Create a basic list view.
Option B — /summarizeCreate .claude/commands/summarize.md Give me a summary of $ARGUMENTS. Read the file, then explain: - What it does (one sentence) - Its public API (functions/routes it exports) - What depends on it and what it depends on - Any code smells or issues you notice
Option C — /changelogCreate .claude/commands/changelog.md Generate a changelog entry for the recent changes. Run git log --oneline -10 and summarize the changes in a user-friendly format with sections for Added, Changed, and Fixed.

After they create it: "Now use it. Type /your-command and see it work."

Common question
"Can I use Claude to write the command?" Yes! "Create a command at .claude/commands/my-thing.md that does X" works great. Commands are just markdown files.
Demo Skills: Teaching Claude Your Rules 12 min

Say: "Commands are things you trigger. Skills are things Claude knows. They're context that Claude reads automatically when it's relevant — your project's coding standards, security rules, architectural decisions."

The difference:

  • CLAUDE.md — loaded every conversation, every time. Put universal rules here.
  • Skills — loaded when relevant. Claude reads the description field to decide if it applies. Put topic-specific guidance here.

Step 1 — Create a coding standards skill:

Promptcreate a file at .claude/skills/coding-standards.md with this content. make sure to include the frontmatter:
--- description: BookBuddy coding standards and conventions --- # BookBuddy Coding Standards ## Naming - Use camelCase for all JavaScript variables and functions - Use kebab-case for file names and URL paths - Database columns use snake_case (SQLite convention) ## Routes - Each resource gets its own file in routes/ - Route handlers must have try/catch with a 500 response on error - Validation happens at the top of each handler before any DB calls - Use parameterized queries (?) — never interpolate user input ## Testing - Use the built-in Node.js test runner (node:test) - No external test frameworks - Test files go in tests/ with the pattern *.test.js

Point out: "The description in the frontmatter is what Claude uses to decide if this skill is relevant. When you ask it to write a route handler, it'll see 'coding standards and conventions' and load the file."

Step 2 — Test that it works:

Promptadd a new route for managing reading lists. a reading list has a name and a list of book IDs.

What to point out:

  • "It used camelCase for variables — that's the skill talking."
  • "It added try/catch to every handler — we didn't ask, but the skill says to."
  • "It used parameterized queries — the security rule from the skill."

Step 3 — Show the "before and after" (if time allows):

Delete the skill, restart Claude (/clear), ask for the same route. Compare the output. "Without the skill, Claude makes its own choices. They're fine, but they might not match your team's standards."

Key idea
Skills are how you scale consistency. One developer writes the coding standards once. Every developer on the team gets them automatically. Claude follows them without anyone having to remember to paste the rules.
Commands vs skills vs CLAUDE.md
CLAUDE.md — always loaded, universal project info (tech stack, build commands, top-level rules).
Skills — loaded when relevant, based on the description field (coding standards, security checklist, deployment guide).
Commands — never auto-loaded, triggered by the user with /name (workflows, checklists, generators).
Exercise Write a Skill 7 min

Say: "Write a skill for something you'd want Claude to always know about this project. Here are some ideas."

Option A — Security checklistCreate .claude/skills/security-checklist.md Write rules about SQL injection prevention, input validation, output escaping, and error message hygiene. Include the frontmatter with a description field.
Option B — Testing conventionsCreate .claude/skills/testing-guide.md Write rules about how tests should be structured: what to name test files, how to set up/tear down the database, what assertions to use, how to organize test cases.
Option C — Let Claude write itlook at the BookBuddy codebase and create a skill file at .claude/skills/architecture.md that describes the project architecture, key patterns, and conventions you can infer from the existing code. include the frontmatter with a description field.

After they create it: "Test it — ask Claude to do something that your skill covers. See if it follows the rules."

Demo Hooks: Automated Guardrails 10 min

Say: "Commands are things you trigger. Skills are things Claude reads. Hooks are things that happen automatically — shell commands that fire when Claude does something. No human in the loop."

Hook events:

  • PreToolUse — runs before Claude uses a tool (can block it)
  • PostToolUse — runs after a tool completes (can add feedback)
  • Notification — runs when Claude sends a notification

Step 1 — Create a simple hook:

"Let's add a hook that checks for stray console.log statements every time Claude edits a file."

Promptcreate a .claude/settings.json file with a PostToolUse hook that runs after Edit or Write tool calls. the hook should grep for console.log in the routes/ directory and warn if any are found (excluding the server startup log).

Show the result:

{ "hooks": { "PostToolUse": [ { "matcher": "Edit|Write", "command": "grep -rn 'console\\.log' routes/ --include='*.js' | grep -v 'BookBuddy running' || true", "description": "Check for stray console.log after file edits" } ] } }

Point out the structure:

  • matcher — which tool triggers this hook (regex pattern)
  • command — the shell command to run
  • description — human-readable label

Step 2 — Trigger it:

Promptadd a console.log("debug test") to routes/books.js at the top of the list handler

"Watch the bottom of Claude's output — after it edits the file, the hook fires and shows the console.log it just added. Claude sees that output and can react to it."

Step 3 — A more powerful hook: lint on save

Concept — don't need to build live{ "hooks": { "PostToolUse": [ { "matcher": "Edit|Write", "command": "npx eslint --fix $TOOL_INPUT_FILE_PATH 2>/dev/null || true", "description": "Auto-format edited files" } ], "PreToolUse": [ { "matcher": "Bash", "command": "echo '$TOOL_INPUT' | grep -qE 'rm -rf|DROP TABLE|--force' && echo 'BLOCKED: destructive command detected' && exit 1 || true", "description": "Block destructive shell commands" } ] } }

Point out:

  • "PreToolUse hooks can BLOCK the action. Exit code 1 = Claude doesn't run the tool. That's a guardrail."
  • "$TOOL_INPUT_FILE_PATH and $TOOL_INPUT are variables the hook receives — the file being edited or the command being run."
  • "This is the same idea as git hooks or CI checks, but it runs during your Claude session, not after."
Watch out
Hooks run on every matching tool call. A slow hook (like running the full test suite on every edit) will make Claude feel sluggish. Keep hooks fast — grep, lint a single file, check one thing. Save the heavy checks for commands like /preflight.
Key idea
Hooks are guardrails that don't require discipline. You don't have to remember to check for console.logs or run the linter — it happens automatically. That's the difference between "we have a rule" and "we enforce a rule."
Talk Putting It All Together 5 min

Say: "Let's see how these three layers compose into a real workflow."

Scenario: A new developer joins the team.

  1. They clone the repo. The CLAUDE.md tells Claude what the project is and how to build it.
  2. They start coding. Skills automatically load the coding standards and security checklist — Claude follows them without being told.
  3. They type /add-route users. The command scaffolds a new route following the team's pattern.
  4. Claude edits files. The hook checks for console.logs and auto-formats the code.
  5. They type /preflight. The command runs the full checklist before they commit.

"Zero onboarding conversation. Zero 'hey, what's our convention for X?' Slack messages. The workflow is in the repo."

When to use which:

I want Claude to... Use
Do a specific task on demand Command
Follow rules without being told Skill (or CLAUDE.md)
Run a check automatically on events Hook
Know the basics about the project always CLAUDE.md

Takeaway: "The real power of Claude Code isn't the one-off prompts — it's building a system that makes your whole team faster. Commands, skills, and hooks are how you get there. And since they're just files in your repo, they evolve with your codebase."

Talk Q&A 3 min

Common questions:

  • "Can I have personal commands that aren't in the repo?" — Yes, put them in ~/.claude/commands/ for global commands available in every project.
  • "How many skills is too many?" — Claude uses the description to decide relevance, so more skills don't hurt. But keep each one focused — one topic per file.
  • "Can hooks modify Claude's behavior?" — Hooks can print output that Claude sees and can react to, and PreToolUse hooks can block actions. But they can't change Claude's instructions directly.
  • "Do these work in VS Code / the IDE extension?" — Yes, same files, same behavior. The .claude/ directory works across all Claude Code interfaces.
