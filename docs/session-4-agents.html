The Idea

Real development has phases — write the code, test it, review it. Each phase needs different expertise and different constraints. This session is about writing a single command that runs a multi-phase pipeline, where each phase has its own role definition.

No SDK. No framework. Just markdown files orchestrating markdown files.

Talk The Mental Model 10 min

Say: "In the earlier sessions you talked to one Claude doing one thing. But real development workflows have phases — write the code, test it, review it. Each phase needs different expertise and different instructions. This session is about writing a single command that runs a multi-phase pipeline, where each phase has its own role definition."

Three concepts, one sentence each:

Roles

A skill file that defines how Claude should behave for a specific job. Just markdown in .claude/skills/roles/.

Orchestrator

A slash command that sequences the roles: "do X as the code writer, then Y as the test writer, then Z as the reviewer."

Context Passing

Phases communicate through the code on disk and git diff. No plumbing required.

The "aha" moment:

"You already know how to write commands and skills from the last session. An agent pipeline is just a command that references skills in sequence. There's no new syntax, no SDK, no framework — it's markdown files orchestrating markdown files."

Why roles matter — separation of concerns:

"When you ask Claude to 'build a feature and write tests and review it,' everything bleeds together. The code writer adds tests mid-feature. The reviewer starts refactoring. Roles are constraints — the code writer doesn't write tests, the test writer doesn't touch production code, the reviewer doesn't add features. That's what makes the pipeline reliable."

Where everything lives:

.claude/ ├── commands/ │ └── build-feature.md # the orchestrator └── skills/ └── roles/ ├── code-writer.md # role definition ├── test-writer.md # role definition └── reviewer.md # role definition
Framing
"Think of it like a code review pipeline at your company. A developer writes the code, a CI system runs the tests, a reviewer checks the PR. Same phases — we're just defining each role in a file and letting one command run the whole thing."
Demo Build /build-feature Live 15 min

Say: "Let's build a pipeline from scratch. We're going to create three role files and one command that sequences them. The task: add a /health endpoint to BookBuddy."

Step 1 — Create the role skills (do this live):

Promptcreate the directory .claude/skills/roles/ and create three files in it: 1. code-writer.md with this content: --- description: Code writer role — implements features following existing patterns --- # Code Writer Role You are a code writer. Your ONLY job is to write production code. ## Rules - Follow existing patterns — look at how other routes are structured before writing yours - Use the same error handling style as existing routes - Never write tests — that's not your job - Never refactor unrelated code — stay focused on the task 2. test-writer.md with this content: --- description: Test writer role — writes tests for new code using Node test runner --- # Test Writer Role You are a test writer. Your ONLY job is to write tests. ## Rules - Run git diff to see what changed — write tests for THOSE changes, nothing else - Use the built-in Node.js test runner (node:test), no external frameworks - Cover the happy path and at least one edge case - Never modify production code — if something looks broken, note it but don't fix it 3. reviewer.md with this content: --- description: Code reviewer role — reviews changes for bugs and pattern violations --- # Reviewer Role You are a code reviewer. Your ONLY job is to review. ## Rules - Look at all uncommitted changes (git diff) - Check for: bugs, missing error handling, inconsistency with existing patterns - If you find issues, fix them - If the code is clean, say LGTM and summarize what was built

Point out two things as Claude creates them:

  • "Each role says what it does and what it doesn't do. The code writer never writes tests. The test writer never touches production code. Constraints are the whole point."
  • "These are just skill files — same thing we built in the last session. Nothing new here yet."

Step 2 — Create the orchestrator command:

Promptcreate .claude/commands/build-feature.md with this content: Build this feature: $ARGUMENTS Follow this pipeline — complete each phase fully before moving to the next: Phase 1 — Code Writer: Read .claude/skills/roles/code-writer.md and follow those rules. Implement the feature. Phase 2 — Test Writer: Read .claude/skills/roles/test-writer.md and follow those rules. Write tests for what you just built. Phase 3 — Validate: Run the tests with `node --test`. If any fail, fix the issue and re-run. Repeat up to 3 times. Phase 4 — Reviewer: Read .claude/skills/roles/reviewer.md and follow those rules. Review all changes.

Point out:

  • "$ARGUMENTS — whatever you type after /build-feature gets inserted here."
  • "Each phase reads a different skill file. That's the role switch."
  • "The 'up to 3 times' in Phase 3 is a circuit breaker. Without it, Claude could loop forever."

Step 3 — Run it:

Type in Claude Code/build-feature add a GET /health endpoint that returns JSON { status: "ok" }

As it runs, narrate:

  • "Phase 1 — it read the code-writer role, looked at the existing routes for patterns, and wrote a minimal health route. Notice it didn't add tests."
  • "Phase 2 — it read the test-writer role, ran git diff to see what changed, and wrote tests for just the health endpoint."
  • "Phase 3 — it's running the tests. If they pass, it moves on. If not, it'd fix and retry."
  • "Phase 4 — it read the reviewer role and is checking the diff for issues."
Key moment
"One command. Four phases. Three different roles. The task was 'add a health endpoint' and we got implementation, tests, validation, and review — all from typing one line."
If it takes too long
The /health endpoint is intentionally tiny. If Claude is slow, that's fine — narrate what it's doing at each phase. The audience is watching the structure, not the speed.

Demo Richer Roles, Context Passing, Failure Handling 15 min

Say: "The pipeline we built works, but the roles are pretty bare. Let's make them smarter. Three patterns that make pipelines reliable."

Pattern 1: Richer role skills (5 min)

"Let's upgrade the code-writer role from 'follow existing patterns' to something with real preparation steps."

Promptupdate .claude/skills/roles/code-writer.md — add a "Before you write" section: - Read the existing route files to understand the project's patterns - Check the database schema in db/schema.sql - Look at how errors are handled in existing routes And an "After you write" section: - Run the app with npm start and verify your endpoint works with curl - Stage your changes with git add

Point out: "The role isn't just 'be a code writer.' It tells Claude how to prepare before writing and how to verify after. Each role becomes a checklist."

Pattern 2: Context passing via the filesystem (5 min)

Say: "How does the test writer know what the code writer built? How does the reviewer know what changed?"

"It checks git diff. The filesystem IS the context bus. You don't need to pass variables between phases — the code on disk is the shared state."

Show this addition to test-writer.md## Before writing tests Run `git diff` to see exactly what changed. Write tests for THOSE changes, nothing else. Don't test existing code that wasn't modified.

Point out: "No special plumbing. No passing objects between agents. Phase 2 reads what Phase 1 wrote. Phase 4 reads what Phase 2 and 3 changed. Git is the communication layer."

Key idea
The filesystem is your context bus. Every phase reads the current state of the code and the diff. That's all you need.

Pattern 3: Failure handling and circuit breakers (5 min)

Say: "What happens when the tests fail? Right now our command says 'repeat up to 3 times.' Let's make that smarter."

Upgrade the validate phase in build-feature.mdPhase 3 — Validate: Run the tests with `node --test`. If any fail: a. Read the failure output carefully b. Determine if it's a production code bug or a test bug c. Fix the right one (go back to the code-writer role if it's production code, or the test-writer role if it's a test bug) d. Re-run the tests e. Repeat up to 3 times. If still failing after 3 attempts, stop and report: what's failing, what you tried, and what you think the root cause is.

Point out:

  • "Step (b) is the key — it's not just 'try again,' it's 'diagnose first.' Is the code wrong or is the test wrong?"
  • "Step (c) switches back to the right role. The fixer follows the code-writer constraints. Roles persist even in the retry loop."
  • "The '3 times then stop and report' is the circuit breaker. Always set a bound. Without it, you burn tokens and get nowhere."
Common mistake
Retry loops without a bound. "Fix and re-run until tests pass" sounds reasonable but can run indefinitely if there's a deeper issue. Always cap retries and have a fallback behavior (stop and report).
Exercise Build a Richer Role, Wire It In 10 min

Say: "Take the command you built in Exercise 1. Pick one of the roles and make it richer — add preparation steps, constraints, and self-verification. Then test your pipeline on a small task."

Things to try:

  • Add a "Before you start" section — what should the role read first? What patterns should it follow?
  • Add an "After you're done" self-check — run tests, curl the endpoint, check the diff looks clean
  • Add a scope constraint — "don't modify files outside of routes/"
  • Add a retry rule to the validation phase with a bound

Small tasks to test on (cheap, fast):

  • Add a GET /books/:id/reviews endpoint
  • Add input validation to an existing route
  • Add a 404 handler for unknown routes
What to look for
Does the richer role actually change Claude's behavior vs the bare-bones version? The "Before you start" section should make Claude do research before writing. The constraints should prevent it from going off-script.
Hands-on Open Experimentation 25 min

Say: "You've got 25 minutes. Build something. Here are some ideas at different levels — or go completely off-script."

Level 1 — Extend your pipeline:

  • Add a fifth phase: a documentation writer that updates the README based on what was built
  • Add a security checker role between the code writer and test writer
  • Make the reviewer more specific — give it a checklist of things to look for

Level 2 — Build a new pipeline from scratch:

  • /investigate — researcher reads the codebase and writes a summary, analyst identifies top 3 improvements, planner writes a plan
  • /security-audit — scanner checks for common issues (SQL injection, missing validation), reporter writes a summary with severity ratings

Level 3 — Bring your own project:

  • If you have a repo on your laptop, point Claude at it and build a pipeline that fits your actual workflow
As you circulate
Common sticking points: roles that are too vague ("be a good reviewer" — help them add specific constraints), commands that try to do too much in one run (help them scope down), and forgetting the circuit breaker on retry loops.

At 10:35 (5-minute warning): Ask 1–2 people to share what they built. What roles did they define? What constraints worked well? Did anything surprise them?

Talk Wrap-up 5 min

Say: "Let's recap. You built agent pipelines today with zero frameworks, zero SDKs — just markdown files."

Three takeaways:

  1. Roles are constraints, not capabilities. The power isn't telling Claude what to do — it's telling it what NOT to do. A code writer that doesn't write tests. A reviewer that doesn't add features. Separation of concerns via prompt.
  2. The filesystem is your context bus. Phases communicate through the code on disk and git diff. No variable passing, no state management, no plumbing.
  3. Always add a circuit breaker. Any loop needs a bound. "Retry up to 3 times, then stop and report." Without it, you burn tokens and get nowhere.

Close: "Everything you built today is committed to your repo. Your teammates can use these pipelines tomorrow without knowing how they work — they just type /build-feature and go."

Where it goes from here
"These patterns — roles, orchestration, context passing, failure handling — are the same patterns you'd use if you moved to the Claude Code SDK or built a full agent framework. The difference is scale, not shape. Start with markdown. Graduate when markdown isn't enough."
